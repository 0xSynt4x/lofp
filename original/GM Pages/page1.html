Basic scripting

The script language compiles plain ASCII text files: document formats from word processors cannot be used. A text editor is recommended for creating script files. The blocks of text are read into the compiler, ignoring leading tabs and spaces, and interpreted into a machine format that Legends of Future Past interprets as players interact with the game. Lines of text in the script file that begin with a semicolon are ignored as "comment lines." It is not case-sensitive.Nouns and Adjectives The most simple element of the script language is the ability to define all the nouns and adjectives that Legends will recognize. This is done with the NOUNDEF and ADJDEF commands, which take the format NOUNDEF [noun #] [noun name]. 
                Examples:

                        NOUNDEF 0 agate
                        NOUNDEF 1 alexandrite
                        NOUNDEF 2 amethyst
                        NOUNDEF 3 altar
                        NOUNDEF 4 amulet
                        NOUNDEF 5 anklet
                        NOUNDEF 6 aquamarine
                        NOUNDEF 7 arch

                        ADJDEF 0 acid
                        ADJDEF 1aelfen
                        ADJDEF 2 alexandrite
                        ADJDEF 3 alkyline
                        ADJDEF 4 amazing
                        ADJDEF 5 alzyron
                        ADJDEF 6 amethyst
                        ADJDEF 7 ancient

Defining items

Before an item can be placed in your room description they must be defined. By capturing a @list of items or @gloss for a particular item name you can determine if what you want is already in the system. Even with the large amount of items already available you will find it necessary, at some time, to script a new item.

You will begin all item definitions with the INUMBER command, which takes the format INUMBER #. Before you assign the item you are making a number you must be assigned a block of blank item numbers with which to work. Always use the lowest item number you are assigned and work upwards.

The first thing that every item needs is a name. To give the item a name, use the NAME command which takes the format NAME [noun #]. Now that you have defined the INUMBER and the NAME of the tiem there are a variety of other things that must now be defined as well.

WEIGHT [pounds]

This is simply the weight of the item in pounds. Any object with a weight of 1000 or more is not moveable by the players (they can't pick it up and take it with them, etc.). Doors, trails, and other such permanent items should be defined as having a weight of 1000.

VOLUME [amount]

This is an abstract value representing the amount of space that item will take up. It is not based upon any type of "real" volume. This value limits what can be put in what containers, e.g. a great sword can not fit in a lock box. The rough valumne equivalent are as follows: 
1    Keys, arrows, daggers, gems or smaller
2    Hand axe
3    Light crossbow
4    Club
5    Short sword, light armor
6    Short bow, scimitar, long sword
7    Greatsword
8    Long bow, polearm, full armor
9+   Very large objects, probably to big to carry

SUBSTANCE [substance type (see below)]

This describes the type of material the object is made of. If the exact substance is unavailable, then choose the thing that is closest in hardness and durability. If an object is made of more than one material (e.g., a hand axe is made of a metal head and a wooden shaft) chose a substance type which is an average of the two types. If left undefined, the substances is assumed to be INDESTRUCTRABLE. The substance types are: 

INSUBSTANTIAL   Gas and liquids
PAPER           Scrolls, letters
CLOTH           Clothing, ropes
BRITTLE         Glass bottles, gems
WOOD            Anything made of wood
STONE           Primitive stone tools, harder gems, etc.
SOFTMETAL       Softer metals (silver, gold, tin, brass)
HARDMETAL       Harder metals (iron, steel, titanium)
INDESTRUCTABLE  Anything extremely hard (Alzyron or magical metals) or otherwise
                indestructable for reasons of game integrity (Doors,
trails, etc.)

ARTICLE [a/an/the/some/your]

If left undefined "ARTICLE A" is assumed. This defines how the game refers

CONTAINER [in/on/under/behind]

Defines the object as a type of container, i.e., any object that can have something put on, in, under or behind it. No object may contain items in two different ways (e.g., a bed could have things able to be put on it or under it, but not both).

INTERIOR [amount]

This value is only necessary for items that will act as containers of some sort. It defines how much the container can hold, it may contain uup to [amount] in total volume of items, as defined by the VOLUMNE definitions of the contained items. Note howeve,r that the container's VOLUME value itself also comes into play when the game is determining whether or not an object can fit in the container: if the item to be contained is equal to or greater than the container's VOLUME, it won't fit.

PARAMETER1 [value]

Defines a value for special parameter 31. It is unnecessary for most items. Fo liquids, it is the amount of drinks that can be taken from it before it runs out (and if over 100, it is treated as unlimited). For weapons, it defines the maximum damage the weapon will do and for armor it defines the percentage of damage that the armor absorbs. Keep in mind the values given for other such items when deciding on what this value should be. The usefulness of this parameter may be expanded in the future.

PARAMETER2 [value]

Defines a value for special parameter2. It is unnecessary for most items. For missile weapons, it defines the item number to be used as its ammunition type. For ammunition, it defines the number of individual units of ammunition are included in a bundle.
(The usefulness of this parameter may be expanded in the future.)

PARAMETER3 [value]

This is similar to PARAMETER1 and PARAMETER2 on items. It will be used for more complex item archetypes in the future.
