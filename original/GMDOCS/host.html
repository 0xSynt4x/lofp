Basic scripting

The script language compiles plain ASCII text files: document formats from word processors cannot be used. A text editor is recommended for creating script files. The blocks of text are read into the compiler, ignoring leading tabs and spaces, and interpreted into a machine format that Legends of Future Past interprets as players interact with the game. Lines of text in the script file that begin with a semicolon are ignored as "comment lines." It is not case-sensitive.

Nouns

The most simple element of the script language is the ability to define all the nouns and adjectives that Legends will recognize. This is done with the NOUNDEF and ADJDEF commands, which take the format NOUNDEF [noun #] [noun name].

Examples:
  • NOUNDEF 0 agate
  • NOUNDEF 1 alexandrite
  • NOUNDEF 2 amethyst
  • NOUNDEF 3 altar
  • NOUNDEF 4 amulet
  • NOUNDEF 5 anklet
  • NOUNDEF 6 aquamarine
  • NOUNDEF 7 arch
  • ADJDEF 0 acid
  • ADJDEF 1aelfen
  • ADJDEF 2 alexandrite
  • ADJDEF 3 alkyline
  • ADJDEF 4 amazing
  • ADJDEF 5 alzyron
  • ADJDEF 6 amethyst
  • ADJDEF 7 ancient

FORAGEDEF [terrain] [item#] [adjective#] [ratio] [val2> [val5]

This is a definition that states that item# can be found in a room of type [terrain]. The "ratio" variable determines the likelihood of finding this particular item relative to anything else in the area (for example, if one item has a ratio of 10 and another has a ratio of 20, then the second item will be found twice as often as the first). The room terrain for [terrain] is limited to FOREST, MOUNTAIN, PLAIN, SWAMP and JUNGLE. Items should only be forageable if they are prevalent throughout a particular terrain type and if they are generally available throughout all seasons.

MINDEF [item#] [adjective#] [grade] [copper value] [val2]

This is a definition that states that item# with adjective# is a mineable item of "grade" (the latter of which may be 'A', 'B', or 'C', in descending value).

Defining items

Before an item can be placed in your room description they must be defined. By capturing a @list of items or @scanning for a particular item name you can determine if what you want is already in the system. Even with the large amount of items already available you will find it necessary, at some time, to script a new item.

You will begin all item definitions with the INUMBER command, which takes the format INUMBER #. Before you assign the item you are making a number you must be assigned a block of blank item numbers with which to work. Always use the lowest item number you are assigned and work upwards.

The first thing that every item needs is a name. To give the item a name, use the NAME command which takes the format NAME [noun #]. Now that you have defined the INUMBER and the NAME of the tiem there are a variety of other things that must now be defined as well.

ORE

Ore can be sold or it can be smelted by a character with weaponsmithing skill and then worked into the form of a weapon. PARAMETER2 defines the new item archetype created when the ore is refined. The VAL2 field of an ORE item is transferred to any new item it is worked into. The VAL3 field of an ORE item is its purity, i.e., how much useful metal is contained in the amount of rock. This represents a percentage chance that a smith has of refining any useful metal from the ore. The PARAMETER1 field has multiple meanings. If it equals -1, then no adjective is placed on a new item constructed from the material. If it is zero, then the old adjectives of the material are transferred to the adjective to place on a new item created from the material. The PARAMETER3 field specifies the grade of the ore.

WEIGHT [pounds]

This is simply the weight of the item in pounds. Any object with a weight of 1000 or more is not moveable by the players (they can't pick it up and take it with them, etc.). Doors, trails, and other such permanent items should be defined as having a weight of 1000.

VOLUME [amount]

This is an abstract value representing the amount of space that item will take up. It is not based upon any type of "real" volume. This value limits what can be put in what containers, e.g. a great sword can not fit in a lock box. The rough volume equivalent are as follows:

  • 1-Keys, arrows, daggers, gems or smaller
  • 2-Hand axe
  • 3-Light crossbow
  • 4-Club
  • 5-Short sword, light armor
  • 6-Short bow, scimitar, long sword
  • 7-Greatsword
  • 8-Long bow, polearm, full armor
  • 9+- Very large objects, probably to big to carry

SUBSTANCE [substance type]

This describes the type of material the object is made of. If the exact substance is unavailable, then choose the thing that is closest in hardness and durability. If an object is made of more than one material (e.g., a hand axe is made of a metal head and a wooden shaft) chose a substance type which is an average of the two types. If left undefined, the substances is assumed to be INDESTRUCTRABLE. The substance types are:

INSUBSTANTIALGas and liquids
PAPERScrolls, letters
CLOTHClothing, ropes
BRITTLEGlass bottles, gems
WOODAnything made of wood
STONEPrimitive stone tools, harder gems, etc.
SOFTMETALSofter metals (silver, gold, tin, brass)
HARDMETALHarder metals (iron, steel, titanium)
INDESTRUCTABLEAnything extremely hard (Alzyron or magical metals) or otherwise indestructable for reasons of game integrity (Doors, trails, etc.)

ARTICLE [A|AN|THE|SOME]

If left undefined "ARTICLE A" is assumed. This defines how the game refers to the object, i.e., is it "a sunsword," "some sunsword" or "the sunsword."

CONTAINER [type]

Defines the object as a type of container, i.e., any object that can have something put on, in, under or behind it. No object may contain items in two different ways (e.g., a bed could have things able to be put on it or under it, but not both).

INTERIOR [amount]

This value is only necessary for items that will act as containers of some sort. It defines how much the container can hold, it may contain uup to [amount] in total volume of items, as defined by the VOLUMNE definitions of the contained items. Note howeve,r that the container's VOLUME value itself also comes into play when the game is determining whether or not an object can fit in the container: if the item to be contained is equal to or greater than the container's VOLUME, it won't fit.

PARAMETER1 [value]

Defines a value for special parameter 31. It is unnecessary for most items. For liquids, it is the amount of drinks that can be taken from it before it runs out (and if over 100, it is treated as unlimited). For weapons, it defines the maximum damage the weapon will do and for armor it defines the percentage of damage that the armor absorbs. Keep in mind the values given for other such items when deciding on what this value should be. The usefulness of this parameter may be expanded in the future.

PARAMETER2 [value]

Defines a value for special parameter2. It is unnecessary for most items. For missile weapons, it defines the item number to be used as its ammunition type. For ammunition, it defines the number of individual units of ammunition are included in a bundle. The usefulness of this parameter may be expanded in the future.

PARAMETER3 [value]

This is similar to PARAMETER1 and PARAMETER2 on items. It will be used for more complex item archetypes in the future.

Additional parameters: Any of the following may appear on a line by themselves to define special information about an object. You could use one of them, none of them or any combination of them.

CRAFTABLE

When placed on an item, it makes that item available for crafting by a player craftsman. Be carefull with this. Players are limited as to what craftable items they can actually make as follows: if an item is a weapon, then the player's smithing skill must >= the weapons parameter1 field. If it is a piece of armor, it is limited by one-third the armors weight. If it is a piece of clothing or jewelry it is limited by whatever parameter2 happens to be. NOTE: No more than one item should be craftable per noun.

DYEABLE

This is an item modifier that states that the given item archetype can be died, assuming it has no adjectives in slots #1 or #2.

ENCRUSTABLE

This makes it possible to use the INSET, INLAY, ENCRUST and similar verbs on an item.

DYE

This is an item modifier that can be used with other item definitions. When defined, it allows anyone with Dyer/weaver skill to extract the pigment from the object and use it to color a cloth item. On DYE items, PARAMETER1 defines the #1 adjective and PARAMETER3 defines the #2 adjective to place on an item it is dyed with. A different archetype needs to be made for every item that can dye a different color.

Defining Rooms

Every room has to have certain parameters defined for it. This includes the room number, a name, the terrain, the lighting and a description. These are the parameters that must be included in the script. Later, we will discuss additional modifiers and conditional statements that you can add to 'spice' it up.

Number

The compiler recognizes that you are defining a room when you use the NUMBER command. It takes the format NUMBER [#] where # is the number of the room you are defining.

To give your room a name use the NAME command on a line by itself after defining the number of the room. The format is NAME [text].

Room Descriptions

The description isentered by placing *DESCRIPTION_START on a line by itself, entering the text for the description on following lines and ending it with *DESCRIPTION_END on a line by itself. Warning: Remember to end the description with the proper statement. Also make sure it is correctly spelled. If you forget or mispell the ending statement you will have all of the script that follows included in the room description until it encounters a *DESCRIPTION_END.

Example:

*DESCRIPTION_START
This is a room like many other rooms.
*DESCRIPTION_END

Item Value Fields

Variables may be defined to keep track of numeric values. To define a variable, simply place VARIABLE on a line by itself anywhere in your script.

DIV   | Division
EQUAL | Equal to
MOD   | Modulus division
MUL   | Multiplication
SUB   | Subtraction

VARIABLE TESTVAR
VARIABLE VAR2
EQUAL TESTVAR 3
EQUAL VAR2 7
ADD VAR2 3
MUL TESTVAR VAR2

The above would define two variables, "Testvar and Var2." Testvare is set equal to 3 and Var2 equal to 7. The number 3 is then added to Var2 making it 10. Testvar is then multiplied by Var2.

8 - Undead affecting. This is the only immunity that is default to 0 and it must be set to a special value. When set to 3 it defines a creature as a lesser undead, and includes creatures up to and including a lesser wight. Setting it to 2 defines it as a greater undead such as a mummy, lesser vampire, etc. Setting it to 1 should be reserved for extremely powerful undead such as a lich, a greater vampire, etc.

9 - Susceptability to death criticals. If set to 0 the creature would not receive a fancy death critical. This should be given to any creature that lacks a normal humanoid or animal physiology.

1. Ability to seperate game areas into distinct geographic/economic regions

REGIONDEF  

Defines a "region" number "x" (which may be 0-250).  A region is a set
of rooms that share a number of configurable parameters.  The available
settings are as follows:

  REGIONDEF x WEATHER

  Specifies that weather will occur in this region.

  REGIONDEF x EXTRAPLANAR

  Specifies that your region is extraplanar.  This has a number of
  effects, but the most important is that characters travelling here
  are limited in skill use as per the Legends version notes regarding
  the transcendence skill.

  REGIONDEF x SEASONS

  Specifies that your region has seasons or not.  You still need to
  specify the base temperature for the different parts of the year as shown
  below.

  REGIONDEF x TIME

  Specifies whether your region experiences the passage of time observable
  through the movement of the sun.

  REGIONDEF x WINTERTEMP 
  REGIONDEF x SUMMERTEMP 
  REGIONDEF x SPRINGTEMP 
  REGIONDEF x AUTUMNTEMP 

  Defines the base temepature in farenheit for a given region.  This
  base temperature is used to determine a region's environmental
  temperature on a daily basis.  The formula used for the actual
  temperature is:

         actual_temp = base_temp + rnd(15) + rnd(15);

  So if you put "20" for the base temperature, the possible range of
  actual temperatures for your region would be 22-50, with the most
  likely average temperature being around 36.

  REGIONDEF x PRECIPITATION 

  Precipitation level is simply a relative scale measuring how much
  rain/snow will fall in a given region.  A precipitation level of 0
  would be "normal" (equal to the precipitation level used in the game
  so far).  The closer to 10 you get, the less likely it is to create
  precipitation at any time and the more likely it is for existing
  precipitation to cease.  If you specify a negative number, it will be
  increasingly likely to have precipitation at any time.  If you specify
  a precipitation of 10, it would probably never rain.

  REGIONDEF x TREASURE 

  This is only valid in regions 0-10.  When this is specified, it allows
  you to state what percent of "local currency" (as defined with a
  MONDEYDEF with the same money definition number as the region number)
  appears in the pockets of monsters genned in the region.  Just state
  "100" to have a 100% chance if you want to have local currency only.
  Non-local currency is simply the common gold, silver and copper piece.

  REGIONDEF x DEPART 

  Specifies the room number a player will be moved to if they use the
  DEPART command while dead.  If room == 0, then it is impossible for
  them to depart in the specified region.

  REGIONDEF x TELEPORT ALLOWED    or...
  REGIONDEF x TELEPORT DISABLED

  Specifies whether you region permits teleportation -within- your
  region or not.  Teleportation is never permitted between regions.

  REGIONDEF x SUMMONING ALLOWED  or...
  REGIONDEF x SUMMONING DISBABLED

  Specified whether you want to allow summoning spells to function
  within your region.

  REGIONDEF x WELLOFLIFE ALLOWED   or...
  REGIONDEF x WELLOFLIFE REVERSED  or...
  REGIONDEF x WELLOFLIFE DISABLED

  Specified whether the Well of Life spell works or not, or if you'd
  like its effect reversed (so that it functions as Raise Undead).

  REGIONDEF x FIRE 

  Specifies the percent effectiveness of fire-base spells and other
  effects in region x.  For example, a "REGIONDEF 15 FIRE 50" would
  make it so that fire spells would do only half damage in region 15.

  REGIONDEF x COLD 

  Specifies the percent effectiveness of cold-base spells and other
  effects in region x.

  REGIONDEF x ELECTRIC 

  Specifies the percent effectiveness of electrical and energy spells
  and other effects in region x.

  REGIONDEF x ENCHANTMOD 

  Specifies the penalty to apply against enchanted weapons in region x.
  It will never penalize an enchanted weapon to lower than a bonus of 0.
  For example, you could make it so that all weapons are dropped in
  bonus by 20 to a minimum of 0 in region 15 by doing
  "REGIONDEF 15 ENCHANTMOD 20".  Then, +20 weapons would become +0,
  +15 would become +0, +25 would become +5, etc.


REGION 

When inserted in a room definition block, this defines the region of
the world that the room belongs to.  If not specified, the default is 0.
Defining a region has the following effects:


  - Temperature, weather, passage of time, spells, etc. are modified
      as per the REGIONDEF for the region.
  - Characters cannot teleport or scry characters in another region.
  - Regions numbered between 0-9 are a special type;  a room in a
    region between 0 and 9 use a money type the same as their region
    number.  For example, if you've got a room in region 5, and you have
    a shop there, they will use money type 5 (see below).

    * * * YOU MUST HAVE A REGION NUMBER ASSIGNED TO YOU * * *


2. Ability to define new units of currency for new regions


MONEYDEF    

  Defines a type of money.  "X" is a number from 1 to 9.  This number
  is equivalent to a type of money native to a given region number 1-9
  (as above).  "Name" is the singular name of the currency.  "Exchange
  rate" is a number from 1 to 100, specifying the number of units of
  this new type of currency that are roughly equivalent to one gold
  piece (of base currency).  The "exchange rate" is used in only two
  places at this time:  the cost of getting healed by a physician, and
  the amount of treasure in this currency unit generated by a monster
  that carries this type of currency relative to standard (base)
  coin types.  The "item#" field specifies the actual item archetype
  to use when creating this type of money in tangible form.

  Here is an example of a new money type now in use:

  MONEYDEF 2 kragenmark 12 1869

  (This defines a unit of currency called the kragenmark, used in
   region #2, of which 12 equals one gold piece;  item number 1869
   is the "physical" item archetype to be used when creating this
   type of money in tangible form)

    * * * DON'T DEFINE A MONEY TYPE WITHOUT AN ASSIGNMENT * * *

  After you create a unit of currency, you still need to make a new
  item archetype and noun for the currency if you want people to be
  able to get it from treasure, pick it up, drop it, etc.  Just create
  a "MONEY" type item.  The PARAMETER2 field specifies the unit of
  currency the item is for.  Example:

  NOUNDEF 1543 kragenmark
  ;
  INUMBER 1869
  NAME 1543
  MONEY
  WEIGHT 1
  VOLUME 1
  PARAMETER2 2
  ARTICLE SOME

3. New room modifiers

HEALER

  Defines a room as a place a character can get healed by a physician
  by sitting down.  There should only be one of these per region
  number 0-9, or else it won't know what money to use.  The only
  region you can go into 'debt' in with the healers is region 0.

BANK

  Defines a room as a place a character can perform banking.  There
  should only be one bank per region number 0-9, or else it will mix
  their money up between banks.  Each bank is limited to one type of
  currency only.  Characters may have a bank account for each currency
  type, but they can only have a safety deposit box at one bank at
  a time (due to system limitations).

4. New variables

SUBMITTING              Equals 1 if the character is submitting
DEPARTROOM              Equals the last room the character departed
                        from, or 0 if they have not departed recently.
REGION                  Equals the number of the region the character
                        is presently standing in.

Legends version 4.80 features some things including the major
psionics update.  Please see the version notes on the LEGENDS

I've added an @psi command for granting/revoking psionic disciplines.
It works similarly to @grantspell.

The following are the new psionic script commands for monsters:


PSIRESIST 
Defines the creature's psionic resistance.  If you don't specify
any, it will default to the same as its normal magic resistance.


PSI 
The number of psi points the creature has.


DISCIPLINE 
Gives the creature the psionic discipline # you specify.


PSIUSE 
Specifies a percentage (1-100) of the time that this creature will
opt to use psionics in preferences to other attack modes.


PSISKILL 
The total chance that the psionic creature has when invoking a
psionic power.  Powerful psionic creatures should have well over
a 100 so as to defeat a player's natural resistance to psionics.


PSILEVEL 
The creature's effective skill level in the Mind over Matter skill.
This is used for determining the effectiveness of Cryokinetics,
Pyrokinetics, Electrify and Kinetic Thrust.

How to design a good psionic creature:

Psionic creatures should always have at least one psionic attack
discipline for overcoming a psionicist's defense modes;  it should
also have a defense mode of its own.  Then just give it whatever
you think it should have.  It will probably take some testing to
find out what monsters can use well.  I've tried to build in
some intelligence for monsters so that they'll make wise selections
from the disciplines you give them based on their situation.

One difference with psionic creatures v. magic creatures:  if you
give them a defense discipline (be it a mental defense or the
physical defenses), they get genned with those disciplines already
active.  Make sure you give psionic monsters enough psi to maintain
whatever they have.

Don't give monsters non-battle psionic disciplines (like levitation,
flight, strengthen steel, etc.).  Don't give them Domination either,
because they aren't smart enough to make use of it properly (at this
time anyway).

Virtual Cafe @verbs

@activate

This command allows you to activate a "monster" that you have @GENMON'ed.

(See @GENMON, @ZAP and @SEDATE)

@additem [archnum]

This command adds an item into the room you are in.

(See @GLOSSARY)

@announce [MODE] [MESSAGE]

This command allows you to announce a message to patrons in the Cafe. There are 5 different modes, but only 2 work in the Cafe. They are 1 (GLOBAL) and 2 (ESP/Mindlink)

@answer

This command allows you to answer an ASSIST by removing the ASSIST from the queue and teleporting you to the room of the player who issued the ASSIST.

@close [item name]

This command simply closes an item. It DOES NOT echo that you closed the item to anyone else who is in the room.

@compile

This command USED TO allow you to compile scripts. It no longer works.

(See @OLDCOMP)

@delete [item name]

This command allows you to delete an item.

@echoplr [job #] [text]

This command allows you to echo text to a Cafe patron.

(See @EXCLUDE, @LWHO, @SND, and @WHISPER)

@editem [item name]

This command allows you to edit the below parameters on an item.

Adjective #1
Adjective #2
Adjective #3		 These 3 can take adjective NUMBERS or NAMES
Val1
Val2
Val3
Val4
Val5
Val6
 These slots are used for various functions in the Cafe, such as room destinations for portals, number of sips in a drink, "money" value of an item, etc. See SCRIPT.DOC for more details.
Cycles to deletion Ignore this. It isn't implemented.
Item Location Type Can be 1 (player), 2 (bank), 3 (contained), 4 (room)
Item Location This is paired with Item Location Type. If the type is a player, you'd enter the job number here. If bank, then you'd enter the region. If contained, then you'd enter the RECORD NUMBER of the item to contain it. And if room, then the room number.
Special Description This allows you to add text to an item if someone LOOKS, READS or EATS/DRINKS it. Simply precede the text with the followly letters: E, R or T (respectively)

Seek ID field This is an advanced function and I will explain it later if you really want to know what it is for. But for now, skip it.
